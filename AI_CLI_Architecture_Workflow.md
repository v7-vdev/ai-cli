# AI CLI Architecture & Safety Workflow

## Overview
The AI CLI has been recently upgraded with a modular, highly secure execution architecture. This document outlines the lifecycle of a command, focusing on how autonomous AI actions are intercepted, verified, logged, and executed.

## Core Architectural Components

1. **Command Parser (`src/utils/commandParser.ts`)**
   Routes user input to the correct modular command (e.g., `/read`, `/write`, `/generate`, `/edit`, `/logs`).

2. **Runtime Context (`src/context/runtimeContext.ts`)**
   A global singleton that holds state for the current session. It initializes and injects the `PermissionManager`, `AuditLogger`, `McpManager`, and `AIProvider`.

3. **Permission Manager (`src/permissions/permissionManager.ts`)**
   Intercepts dangerous autonomous actions. It runs security heuristics (like blocking `rm -rf /` or empty file writes) and prompts the user for interactive consent (`Yes`, `No`, `Always allow for session`).

4. **Audit Logger (`src/logs/auditLogger.ts`)**
   A non-blocking background queue that securely persists all execution events to `.ai-cli/audit.log`. It automatically masks API keys (`[REDACTED_API_KEY]`), deduplicates spam from runaway loops, and rotates files larger than 2MB.

5. **Tool Executor (`src/tools/executor.ts`)**
   The execution engine for the AI. When the AI requests to run a bash command, read a file, write a file, or call an MCP tool, the Executor first halts and queries the `PermissionManager`.

6. **REPL Loop (`src/repl.ts`)**
   The primary interactive terminal. It actively monitors autonomous loops and pauses if the AI executes 5 consecutive tool steps without human input.

## The Workflow Lifecycle

### Step 1: User Input & Routing
When a user types a command like `/edit src/index.ts add a console log`, the `CommandParser` dynamically routes it to `editCommand`.

### Step 2: AI Generation
The AI processes the instruction along with the file contents. If the AI hallucinates or requests dangerous file deletions, length heuristics in the command intercept the output before execution.

### Step 3: Tool Execution & Permission Check
If the AI attempts to write a file or execute a shell command, the `ToolExecutor` pauses execution and invokes `PermissionManager.requestPermission()`.

- **Heuristic Check**: `PermissionManager` statically analyzes the payload. If it contains dangerous vectors (e.g., `rm -rf`), it hard-blocks it instantly.
- **Cache Check**: If the user previously selected "Always allow for session" for this exact payload, the action is instantly approved.
- **User Prompt**: If uncached, the user is presented with a colored CLI prompt to accept, deny, or whitelist the action.

### Step 4: Execution & Audit Logging
Once approved, the `ToolExecutor` runs the action.
Simultaneously, the `AuditLogger` captures the exact event payload. 
- It scans the text via Regex to scrub environment variables and API keys.
- It calculates if the event is spam (more than 3 identical logs in 2 seconds).
- It formats the log level (`[INFO]`, `[WARN]`, `[ERROR]`, `[SECURITY]`) and pushes it into the asynchronous background queue.
- `fs.promises.appendFile` safely writes it to disk without freezing the user's chat loop.

### Step 5: Review & Monitor
The user can actively monitor these logs in real-time by typing `/logs tail` in the CLI. The command spawns a safe stream listener that elegantly shuts down on `Ctrl+C` without killing the overarching CLI process.
